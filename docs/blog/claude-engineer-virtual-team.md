## やりたかったこと

Claude Code 上に、自分の開発チームを再現したかった。

自分の分身、同僚の分身。Slack でのやりとり、PR のレビューコメント、設計書、実装コード——チームメンバーの「考え方の癖」って、実はこういうデータの中に全部残っている。

これを AI に読ませて、「この人だったらどうレビューするか」を再現できたら、チーム開発の質がそのままClaude Code上でも出せるんじゃないかと思って、プラグインを作りました。

https://github.com/watanabe-e-kamakura/claude-engineer

## なぜ「分身」なのか

Claude Code のレビュー、使っている人なら分かると思うんですが、**教科書的**なんですよね。

「N+1 クエリに注意してください」「トランザクションを検討してください」——間違ってはいない。でも、チームの同僚にレビューしてもらうと全然違うコメントが来る。

「これ、前に別プロジェクトで同じ構成にしてハマったから気をつけて」
「ここ来月の機能追加で確実に触るから、先に共通化しとかない？」

**プロジェクトの文脈を知っていて、過去の失敗を覚えていて、チームの方針を理解している。** だから指摘が刺さる。

AI にないのはこれです。毎回ゼロから、どのプロジェクトでも同じようなレビューをしてくる。

じゃあ、チームの文脈を持ったメンバーを AI 上に再現すればいい。

## 分身の作り方

claude-engineer では、`.engineer/team/` にチームメンバーを Markdown で定義します。

### 自分自身の分身

まず自分の分身を作ります。自分の PR レビューコメントや Slack での技術議論を Claude に読ませて、こう頼む。

```
自分のPRレビューコメントとSlackの技術議論を分析して、
team/sato.md にメンバー定義を作って。
レビューの癖、よく出る指摘、設計の好み、判断基準をまとめて。
```

出てくるのはこういうファイル。

```markdown
# 佐藤（リードエンジニア）

## レビューの特徴
- 共通リポジトリへの影響を最初に確認する
- Service への切り出しを好む。Action が太ると必ず指摘する
- データ移行とコード変更は絶対に別 PR にさせる
- 「過剰じゃないか」を常に気にする

## よく出る指摘
- 「この処理、共通側に置いた方がよくない？」
- 「トランザクション張らなくて大丈夫？」
- 「ここ将来増えそうだから最初から拡張可能にしておこう」
- 「テストの前提条件が暗黙的。Factory で明示して」

## 判断基準
- 共通 vs 個別の配置判断：2プロジェクト以上で使うなら共通
- PR サイズ：200行超えたら分割を提案
- やりすぎ判定：今必要ないものは作らない。でも明らかに増えるものは先に対応

## 過去に踏んだ地雷
- Enum を個別リポに書いて、共通化するとき全リポ修正が発生した
- null 許容カラムを追加して、既存データの null が予期しない挙動を起こした
- Job の失敗時にリトライが効かない設計にして、深夜に手動リカバリした
```

**自分のレビューの癖が言語化される。** これがまず面白い。普段無意識にやっていることが明文化されて、「あ、自分こういう観点で見てたんだ」と気づく。

ぶっちゃけ某ローマ字4桁の性格診断より正確な自己分析だと思います。あっちは「あなたは論理的で〜」みたいなふわっとした結果が出るけど、こっちは「お前、トランザクション張り忘れがちだし、Action 太らせる癖あるぞ」って具体的に出てくる。自分の弱点が実績ベースで丸見えになる。

### 実際どうやって作ったか

自分は直近1年分のデータを食わせました。データソースは3つ。

**1. Slack（Slack MCP 経由）**

一番リッチなデータソース。技術議論、設計相談、障害対応のやりとりから「考え方の癖」が出る。

Claude Code に Slack MCP を入れてあれば、こう頼むだけ。

```
自分の直近1年の #dev チャンネルでの発言を検索して、
技術議論・設計判断のメッセージを分析して team/sato.md を作って。
```

Slack MCP がなくても、重要な議論のスレッドをコピペで渡せば同じことはできます。

**2. GitHub PR コメント（gh CLI）**

レビューの癖が一番直接的に出る。何を指摘するか、どこを見ているか。

```bash
# 自分がレビューした PR のコメントを取得
gh api search/issues \
  -f q="is:pr reviewed-by:sato org:my-org created:>2025-04-01" \
  -f per_page=50 \
  --jq '.items[].number' | while read pr; do
    gh api repos/my-org/my-repo/pulls/$pr/comments \
      --jq '.[] | select(.user.login == "sato") | .body'
done
```

これを Claude に渡して「レビューの癖を分析して」と頼む。gh CLI はプラグイン内で既に使っているので追加インストール不要。

**3. 実装コード（gh CLI）**

設計思想・命名規則・抽象化の粒度はコードに出る。

```bash
gh api search/commits \
  -f q="author:sato org:my-org committer-date:>2025-04-01" \
  -f per_page=30
```

Slack + PR コメント + コードの3つを合わせると、かなり精度の高い分身ができます。メールも食わせたい場合は Gmail MCP を入れれば同じ要領で取得できる。

**ポイントは、プラグインがこれらのツールを同梱しないこと。** Slack MCP や Gmail MCP は汎用ツールなので、既に入れている人はそのまま使えばいい。入れていない人は gh CLI + コピペで十分作れる。

### 同僚の分身

同じ要領で、同僚の分身も作れます。データソースは同じ（Slack / PR コメント / コード）。

チームに一人、レビューが鋭い人っていますよね。その人が全部の PR を見るのは現実的に無理だけど、その人の観点を分身として再現すれば、全 PR に目が行き届くようになる。

## 分身がレビューするとどうなるか

claude-engineer には `/engineer:autodev` というコマンドがあって、Issue 番号を渡すと設計→実装→テストを自動で回します。

```
/engineer:autodev #123
```

この中で分身たちが活躍します。

**設計フェーズ** — designer エージェントが設計書を作り、spec-reviewer がチェック。最後にチームメンバーの分身が独立した視点で見る。「佐藤さんだったらここ突っ込むだろうな」というチェックが入る。

**実装フェーズ** — implementer が実装して、code-reviewer がバグ検出。その後チームメンバーがプロジェクト固有の観点で見る。

**テストフェーズ** — test-reviewer がカバレッジ確認。チームメンバーが「この失敗パス、前にハマったやつだからテスト必須」と指摘。

ポイントは、**各フェーズのレビュアーが実装の経緯を知らない**こと。自分で書いたコードを自分でレビューしても甘くなる。だから人格を分けている。

汎用のレビュアー（code-reviewer）がバグやセキュリティの穴を見つけて、分身がチーム固有の文脈で見る。**役割が違うから重複しない。**

## 汎用専門家もいる（でもメインは分身）

プリセットとして DBA、セキュリティ、パフォーマンスの専門家も用意しています。

```
.engineer/team/
├── sato.md         ← 自分の分身
├── yamada.md       ← 同僚の分身
├── guardian.md      ← ガーディアン（デフォルト）
├── dba.md           ← DBA 専門家（任意追加）
└── security.md      ← セキュリティ専門家（任意追加）
```

ただ、**本当に効くのは同僚の分身の方**です。

汎用 DBA は「マイグレーションに down() を書いてください」と言う。
同僚の分身は「前に down() 書き忘れて本番ロールバックできなくなった件あったよね。ここは特に気をつけて」と言う。

文脈がある指摘は刺さる。「なぜそれが重要か」が実体験に紐づいているから、実装者が納得できる。

## 分身が学習していく仕組み

ここが一番伝えたいところです。

分身を作ってレビューさせる。これだけでも十分便利なんですが、もう一歩踏み込んでいます。

**レビュー結果が蓄積されて、チームの共有知になる。**

autodev を回すとレビュー結果がファイルに溜まります。で、`/engineer:review-retrospective` を実行すると、過去のレビュー結果を分析してくれる。

- 同じ指摘が繰り返されている → `principles.md` に昇格を提案
- 新しい失敗パターンが見つかった → `anti-patterns.md` に追加を提案

承認したものだけ反映。次回の autodev では全メンバーがこれを参照する。

こういう流れになります。

1回目：佐藤の分身が「ここ N+1」と指摘
2回目：佐藤の分身がまた「N+1」と指摘
→ retrospective で「N+1 チェック」が principles.md に昇格
3回目：designer が設計段階で eager loading を計画に含める。指摘が出なくなる

**個人の暗黙知が、チームの明文化された知見に変わる。**

分身が指摘する → 繰り返される → ルールに昇格 → 全メンバーが参照

人が辞めても、知見は残る。新メンバーが入ってきても、チームの過去の失敗から自動的に学べる。

## どのプロジェクトでも使える

プラグイン本体は言語・フレームワークに依存しません。

```
claude-engineer（プラグイン）         ユーザーのプロジェクト
┌───────────────────┐          ┌───────────────────┐
│ autodev フェーズ管理 │          │ .engineer/         │
│ レビューエージェント  │    →     │   team/            │
│ フィードバックループ │          │     sato.md        │
│ チーム管理          │          │     yamada.md      │
└───────────────────┘          │   review/          │
  言語・FW 非依存                │     principles.md  │
                                │     anti-patterns  │
                                └───────────────────┘
                                  チーム固有の知見
```

PHP でも TypeScript でも Python でも同じプラグインが動く。**変わるのは team/ の中身だけ。**

Docker や CI みたいなプロジェクト固有の処理は config.yml で hook スクリプトを指定して委譲する形にしてあって、hook がなければスキップされるだけです。

## 試してみる

```bash
claude plugin marketplace add watanabe-e-kamakura/claude-engineer
claude plugin install engineer@claude-engineer
```

プロジェクトで `/engineer` を実行するとセットアップが始まります。

デフォルトではリードエンジニア + ガーディアンの2名体制。ここに分身を追加していく。

```
自分の直近のPRレビューコメント20件を分析して、
team/自分の名前.md にメンバー定義を作って
```

これだけで、次回の autodev から自分の視点でもレビューが入るようになります。

同僚にも「このプラグイン入れて、お互いの分身作ろう」と言えば、チーム全員の視点が AI 上に再現される。

## まとめ

**AI のレビューが物足りないのは、あなたのチームを知らないから。**

claude-engineer でやっていることはシンプルです。

**1. 自分と同僚の分身を作る。**
Slack・PR コメント・設計書・コードから、メンバーの「考え方の癖」を Markdown で定義する。教科書的な AI レビューが、チームの文脈を持ったレビューに変わる。

**2. 分身にレビューさせる。**
autodev の各フェーズで、独立した仮想メンバーがチェックする。役割を分けることで、セルフレビューのバイアスを排除する。

**3. レビューがチームの共有知になる。**
繰り返される指摘はルールに昇格する。個人の暗黙知がチームの資産に変わり、人が入れ替わっても知見が残り続ける。
