# Claude Codeで「完全に独立して動くエンジニア」を作る — APIでは実現できない理由

## AI にコードを書かせたい。でも API だと物足りない

ChatGPT や Claude の API を使ってコード生成を自動化する話、よく聞きますよね。GitHub Actions から API を叩いて PR を自動で作る、みたいなやつ。

やってみると分かるんですが、**出てくるコードがどうにも浅い。**

教科書的には正しいんだけど、自分たちのプロジェクトの文脈を何も知らない。命名規則、設計パターン、過去に踏んだ地雷、チームで共有しているレビュー原則——全部ゼロからスタートする。

結局、API で生成したコードを人間がレビューして修正する。それって自動化できてるんだっけ？という話になる。

## API では使えないもの

なぜこうなるかというと、API は**素の Claude**だからです。

自分たちが Claude Code で育ててきたものが全部使えない。

| | Claude API | Claude Code |
|---|---|---|
| CLAUDE.md（プロジェクトルール） | 使えない | 使える |
| プラグイン・スキル | 使えない | 使える |
| チームの分身（team/） | 使えない | 使える |
| レビュー原則・アンチパターン | 使えない | 使える |
| 過去の設計ドキュメント | 使えない | 使える |
| MCP 連携（Slack, GitHub, DB） | 自分で実装 | 設定だけ |

CLAUDE.md に書いた「共通リポジトリの変更は全プロジェクトに波及する」「Action にロジックを太らせない」みたいなルール。チームの同僚の分身が持つ「前にこのパターンでハマった」という知見。レビューの蓄積から自動生成されたアンチパターン。

これ全部、Claude Code の環境にしかない。API に同じものを再現しようとすると、プロンプトエンジニアリングの沼にハマる。しかも更新するたびに同期が必要になる。

**だったら Claude Code ごと自動で回せばいい。**

## Claude Code を「一人のエンジニア」として独立させる

自分が作った [claude-engineer](https://github.com/watanabe-e-kamakura/claude-engineer) というプラグインには、設計→実装→テストを自動で回す `/engineer:autodev` というコマンドがあります。

```
/engineer:autodev #123
```

これで Issue から設計書を作り、レビューして、実装して、テストまで自動で回る。各フェーズでチームの分身がレビューに入る。過去のアンチパターンを参照して同じ失敗を避ける。

ここまでは前回の記事で書きました。

今回の話は、**これを人間が見ていなくても勝手に回す**方法です。

## /schedule で完全自動化する

Claude Code には `/schedule` という機能があって、cron のように定期実行ができます。しかも Claude Code の環境がそのまま使える。

```
/schedule create --cron "0 9 * * 1-5" "gh issue list --assignee @me --state open --json number,title で未着手のIssueを確認して、優先度が高いものから /engineer:autodev で処理して"
```

これで平日9時に自動で Issue を取りに行って、autodev を回す。

やっていることを図にするとこう：

```
毎朝9時（自動）
  ↓
GitHub Issue を確認
  ↓
未着手の Issue を発見
  ↓
/engineer:autodev #123 を実行
  ↓
Phase 0: 影響分析（チームの impact-rules.md を参照）
Phase 1: 設計（過去の design/ を参照、チームの分身がレビュー）
Phase 2: 実装（anti-patterns.md を参照して地雷を回避）
Phase 3: テスト
  ↓
PR を作成（/engineer:pr）
  ↓
Slack に「PR 作りました」と通知
  ↓
人間がレビューして承認
```

**人間がやるのは最後の「レビューして承認」だけ。**

しかもこのレビューも、チームの分身が既に見た上での PR なので、人間のレビューコストも下がる。

## VPS でやる場合

`/schedule` を使わずに VPS で回したい場合は、`tmux` + `/loop` で実現できます。

```bash
# VPS 上で
tmux new -s claude-agent
mkdir ~/claude-engineer && cd ~/claude-engineer
claude
> /engineer   # 初回セットアップ
> /loop 30m gh issue list --assignee @me --state open で新規Issueを確認して、未着手のものがあれば /engineer:autodev で処理して
```

30分ごとに Issue をチェックして自動処理。`tmux` なので SSH を切っても動き続ける。

ただし `/loop` は7日で自動削除されるので、長期運用なら `/schedule` の方がいい。

## 「育てた知見」が自動化の品質を決める

ここで大事なのは、自動化の品質は**事前にどれだけ Claude Code を育てたか**で決まるということです。

API だとプロンプトの精度がそのまま出力の品質になる。Claude Code だと、日常の開発で蓄積したもの全てが品質に反映される。

具体的にはこういうものが効いてくる：

**team/ の分身**
Slack や PR コメントから作った同僚の分身。自動実行でも「この人だったらここ突っ込むだろうな」というチェックが入る。

**review/anti-patterns.md**
過去のレビューで繰り返された失敗パターン。自動実行でも同じ失敗を避ける。

**review/principles.md**
チームのレビュー原則。自動実行のレビューもこの原則に沿って行われる。

**design/ の過去ドキュメント**
過去に似た機能を設計した記録。自動実行でも「前にこういう設計にしたから合わせよう」と判断できる。

**CLAUDE.md**
プロジェクトのルール。ブランチ命名規約、コミットルール、コーディング規約。

つまり、**普段 Claude Code を使って開発しているだけで、自動化の品質が勝手に上がっていく。** レビューの指摘が `principles.md` に昇格し、失敗パターンが `anti-patterns.md` に追加され、設計ドキュメントが `design/` に蓄積される。

これを API で再現するのは現実的に不可能です。

## 現実的にどこまで任せられるか

正直に言うと、今の段階で「完全放置で PR がマージされる」は厳しいと思っています。

自分が実運用で見えている限界：

- **設計判断のグレーゾーン** — 仕様が曖昧な Issue だと設計の方向を間違えることがある
- **既存コードの暗黙の依存関係** — ドキュメント化されていない依存は見落とす
- **UI/UX の判断** — 「この方がユーザーにとって自然」みたいな判断は弱い

だから現実的な運用はこう：

```
自動: Issue取得 → 設計 → 実装 → テスト → PR作成
人間: PR レビュー → 承認 → マージ
```

**定型的な Issue（カラム追加、画面の項目追加、バッチ修正等）は自動で PR まで出る。** 設計判断が必要な Issue は、設計書まで自動で作って人間に判断を仰ぐ。

これだけでもエンジニアの作業時間は半分以下になる。

## まとめ

**API では「素の Claude」しか使えない。Claude Code なら「育てた Claude」がそのまま動く。**

Claude Code を自動化するということは、CLAUDE.md、プラグイン、チームの分身、レビュー原則、アンチパターン、過去の設計ドキュメント——日常の開発で蓄積した全ての知見を持った状態で、もう一人のエンジニアが独立して動くということです。

やることはシンプルで：

1. **claude-engineer を入れる** — チームの分身とレビュー原則を構築する
2. **普段の開発で使う** — 使うほど知見が蓄積される
3. **`/schedule` で自動化する** — 育てた知見ごと自動で回す

最初から完全自動化を目指す必要はない。まずは Claude Code を日常的に使って「育てる」。育った頃に自動化すると、品質が全然違う。
