# アプローチパターン

## A. タスク分解の型

**大規模機能**
必ずフェーズ分け + 依存順序で分解する:
```
Step 1: 基盤（共有ライブラリ・スキーマ変更）
Step 2: UI + ロジック
Step 3: データ移行（コード変更とは別フェーズ）
Step 4: 追加機能
Step 5: 旧コード削除
```
1サブIssue = 1PR のサイズに収める。

**小規模タスク**
- サブIssue分解しない。単一PR
- 既存パターンの再利用を優先する

## B. ロジックの配置

| 条件 | 置き場所 |
|------|---------|
| リクエストの受け取り→レスポンス返却 | Controller/Handler（薄く） |
| ビジネスルール・計算・状態遷移 | Service/UseCase |
| 複数箇所から呼ばれる可能性がある | Service（必須） |
| 他プロジェクトでも使う可能性がある | 共通ライブラリの Service |
| Model固有の振る舞い | Model のメソッド |

## C. 実装の順序

```
1. 共有ライブラリ（共通ロジック）
2. 共通リポジトリ
3. 個別リポジトリ
4. デプロイ設定
5. テスト
```
共有 → 共通 → 個別 の順番は絶対。

## D. PRの作り方

- Small (<30行): 設定変更、hotfix
- Medium (30-200行): 通常の機能追加・修正
- Large (200行+): 新機能の初期実装

コミットメッセージ: Issue番号をつける、レビュー対応は別コミット。

## E. 優先度判断

| 段階 | 基準 |
|------|------|
| 最優先 | 今スプリントのメイン。ブロッカー |
| 通常 | キューに入っている機能開発・修正 |
| 保留 | 外部の意思決定待ち。理由を明記 |
| 余裕があれば | PoC・調査・改善 |

## F. 意思決定のプロセス

1. まずステークホルダーに聞く
2. Gap分析で差分を可視化
3. 使わない機能は積極的に削除
4. 既存パターンを確認

## G. 調査と実装の分離

調査は調査だけで完結させる。調査結果から実装Issueを別に起こす。

## H. データ移行の鉄則

コード変更とは別フェーズにする:
1. 調査Issue: 影響範囲・件数・エッジケース
2. スクリプトIssue: 移行スクリプト + dry-run
3. 本番実行Issue: 手順書付き

## I. config駆動の設計

動作制御はconfigファイル経由。ハードコードしない。

## J. 非同期処理のパターン

- リトライ制御、遅延調整
- エラーは主処理を止めずにエラーログで検知
- リトライ間隔・最大回数はconfigで管理
