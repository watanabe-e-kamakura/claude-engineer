# レビュー原則

## 1. 設計の責務分離を徹底する
- 処理はあるべき場所に置く
- バリデーション、ビジネスロジック、表示ロジックを混ぜない
- フロントにビジネスロジックを書かない

## 2. エラーハンドリングは影響範囲で判断する
- 補助処理のエラーで主処理を止めない
- ただし無意味なcatchはしない

## 3. null と型の安全性を常に意識する
- nullで既存値を上書きしないか確認する
- パラメータと返り値の型指定を明示する

## 4. マージ前の完成度にこだわる
- TODOを残してマージしない
- 未使用ファイル・検証用コードの削除漏れを確認する

## 5. 共通コードへの影響を常に意識する
- 共通リポジトリの変更は全プロジェクトに波及する
- 削除・変更時は影響範囲を確認する

## 6. 意図と背景を必ず確認する
- 「なぜそうしたか」が分からないコードには質問する

## 7. セキュリティに敏感であれ
- OWASP Top 10を常に意識する
- エラーメッセージで内部情報を漏らさない

## 8. 将来の拡張性を考慮する
- 今は1つでも、将来増えるものは拡張可能な設計にする

## 9. データの整合性を守る
- ORMのイベント・トリガーを活かす実装にする
- トランザクションが必要な処理に漏れがないか確認する

## 10. フレームワークの規約に従う
- フレームワークの慣習に沿い、コードベースの一貫性を保つ

## 11. パフォーマンスを意識する
- N+1、不要なクエリ、大量データ処理のメモリを確認する

## 12. 非同期処理の設計
- Job/非同期処理は失敗時のリカバリを常に考慮する

## 13. フロントの制御はバックエンドで担保する
- フロントのdisable/表示制御だけでビジネスルールを守らない

## 14. 仕様の正確さを追求する
- ロジックが仕様と一致しているか照合する

## 15. リファクタは「一括でやるか、やらないか」
- 部分的なリファクタは避ける

## 16. コミュニケーションを大切にする
- 複雑な仕様確認は口頭を提案する
- 関連する人にメンションして認識合わせをする

## レビュー時のチェックリスト

### 設計・アーキテクチャ
- [ ] バリデーションは適切な場所にあるか
- [ ] ロジックがフロントに漏れていないか
- [ ] 責務の分離は適切か
- [ ] 共通コードへの影響はないか

### データの安全性
- [ ] null更新の危険がないか
- [ ] トランザクションが必要な処理に張られているか

### パフォーマンス
- [ ] N+1になっていないか
- [ ] 大量データはバッチ処理しているか

### 品質
- [ ] 未使用ファイル・検証用コードの残骸はないか
- [ ] エラーハンドリングの戦略は適切か
- [ ] 仕様と一致しているか
- [ ] セキュリティ上の懸念はないか
